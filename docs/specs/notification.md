Notification Engine Engineering Specification

Purpose

Notification Engine merupakan pusat komunikasi antara My Dash dan pemilik VPS. Tujuan utamanya bukan mengirim pesan sebanyak mungkin, melainkan memastikan bahwa setiap pemberitahuan memiliki nilai, dikirim pada waktu yang tepat, melalui media yang tepat, dengan prioritas yang tepat, serta tidak mengganggu pengguna. Notification Engine harus mendukung Dashboard Notification, Toast Notification, In-App Notification, WhatsApp menggunakan Baileys, Telegram menggunakan node-telegram-bot-api, serta Provider baru melalui Plugin di masa depan. Seluruh Notification berasal dari Event Bus, bukan dari Monitoring ataupun modul lain secara langsung. Sistem harus mampu membedakan Information, Success, Warning, Error, Critical, Emergency, dan Security Notification, kemudian menentukan apakah cukup ditampilkan di Dashboard, dikirim ke WhatsApp, Telegram, atau seluruh Provider sekaligus sesuai konfigurasi pengguna. Semua Notification wajib memiliki riwayat, Status Delivery, Retry History, dan Audit sehingga setiap pengiriman dapat ditelusuri kembali.

Notification Lifecycle

Setiap Notification memiliki siklus hidup yang tetap dimulai dari Event Received, Rule Evaluation, Template Rendering, Queue Creation, Priority Assignment, Provider Selection, Delivery Attempt, Delivery Verification, Retry apabila gagal, kemudian Archive ke Notification History. Sebelum dikirim, Engine memeriksa apakah Notification tersebut merupakan duplikat, masih berada dalam Cooldown, telah mencapai Rate Limit, atau termasuk Event yang telah diselesaikan sehingga tidak perlu dikirim kembali. AI wajib menerapkan Notification Deduplication, Grouping, Escalation, Cooldown, dan Quiet Period agar pengguna tidak menerima puluhan pesan ketika satu masalah menghasilkan banyak Event serupa. Seluruh Template mendukung Placeholder dinamis seperti Nama Server, Workspace, CPU, RAM, Disk, Waktu, Severity, Link Dashboard, serta informasi lain yang tersedia sehingga pesan tetap rapi tanpa perlu dibuat manual untuk setiap kasus.

Provider Management and Delivery Strategy

Notification Provider bekerja secara independen melalui Worker yang berbeda sehingga kegagalan WhatsApp tidak memengaruhi Telegram maupun Dashboard Notification. Pengguna dapat mengaktifkan atau menonaktifkan setiap Provider secara terpisah melalui halaman Settings. Untuk WhatsApp, pengguna memasukkan Nomor Owner dan melakukan proses Pairing menggunakan Baileys hingga Status berubah menjadi Connected. Untuk Telegram, pengguna memasukkan Bot Token dan Telegram User ID, kemudian Engine melakukan verifikasi koneksi sebelum Provider dinyatakan aktif. AI wajib mendukung Primary Provider dan Fallback Provider sehingga apabila WhatsApp gagal dalam batas Retry tertentu, Notification dapat dialihkan ke Telegram apabila pengguna mengizinkannya. Seluruh Delivery dicatat lengkap beserta Latency, Retry Count, Provider Response, Error Code, dan Delivery Status.

Notification Rules and Custom Threshold

Pengguna dapat membuat Rule Notification sendiri melalui Dashboard. Contohnya CPU di atas delapan puluh lima persen selama lima menit, Memory di atas sembilan puluh persen, Disk tinggal sepuluh persen, Redis tidak merespons, PostgreSQL Restart, Tunnel Disconnect, GitHub Action gagal, Backup berhasil, Backup gagal, Login baru, Session kedaluwarsa, Plugin Error, AI Timeout, Queue menumpuk, atau Health Score turun di bawah nilai tertentu. Setiap Rule memiliki Threshold, Durasi, Prioritas, Provider Tujuan, Jam Aktif, Cooldown, Escalation, Auto Resolve, serta Template yang dapat disesuaikan. Selain Rule buatan pengguna, My Dash juga menyediakan Rule bawaan yang telah dioptimalkan sehingga Dashboard langsung memberikan perlindungan dasar sejak instalasi pertama tanpa konfigurasi tambahan.

Reliability, Performance, and Security

Notification Engine harus mampu menangani ribuan Notification per menit menggunakan Queue yang terpisah dari proses utama Dashboard. Pengiriman dilakukan secara Asynchronous sehingga Dashboard tidak pernah menunggu Provider selesai merespons. Retry menggunakan Exponential Backoff dengan batas maksimum yang dapat dikonfigurasi, sedangkan Notification yang terus gagal dipindahkan ke Dead Letter Queue untuk dianalisis lebih lanjut. Seluruh Credential Provider disimpan secara aman dan tidak pernah dikirim ke AI maupun ditampilkan di Dashboard. AI wajib memantau Queue Length, Delivery Rate, Retry Rate, Failed Delivery, Average Delivery Time, Provider Availability, serta Notification Throughput sehingga kesehatan Engine dapat diamati secara Realtime. Tujuan akhirnya adalah memastikan setiap Notification penting sampai kepada pengguna secepat mungkin tanpa membebani sistem.

Acceptance Criteria

Notification Engine dianggap memenuhi spesifikasi apabila seluruh Notification berasal dari Event Bus, mendukung Dashboard, WhatsApp, Telegram, serta Provider masa depan melalui arsitektur Plugin, memiliki Queue, Retry, Deduplication, Cooldown, Escalation, Delivery Verification, History, Audit, dan Template dinamis yang dapat disesuaikan pengguna. Sistem harus mampu menangani lonjakan Event, menghindari Spam Notification, mempertahankan performa pada beban tinggi, serta tetap berfungsi penuh meskipun salah satu Provider mengalami kegagalan atau tidak tersedia.
